# 1. Instruksi Utama untuk Coding Agent

Baca seluruh dokumen ini sebelum membuat atau mengubah kode.

Bangun aplikasi sesuai kebutuhan yang tertulis dalam PRD ini. Jangan menghilangkan fitur utama, mengubah arsitektur tanpa alasan kuat, atau menambahkan fitur besar yang tidak diminta.

Apabila terdapat detail kecil yang belum disebutkan, gunakan keputusan teknis yang paling sederhana, aman, modern, dan mudah dirawat. Jangan menghentikan pengerjaan hanya karena detail minor belum tersedia.

Sebelum mulai implementasi:

1. Periksa struktur project dan dependensi yang sudah tersedia.
2. Buat rencana implementasi bertahap.
3. Tentukan migration, model, controller, service, route, komponen, dan halaman yang diperlukan.
4. Kerjakan fitur secara bertahap dari fondasi hingga tampilan publik.
5. Pastikan aplikasi tetap dapat dijalankan pada setiap tahap.
6. Jalankan formatter, type checking, migration, seeder, build, dan test setelah implementasi.
7. Jangan hanya membuat tampilan statis. Seluruh konten utama harus terhubung ke database dan dashboard admin.
8. Jangan menyimpan konten profil, proyek, pengalaman, skill, sertifikat, dan teks halaman secara permanen di komponen React.
9. Gunakan data dummy melalui Laravel Seeder agar seluruh halaman langsung memiliki konten dan dapat diedit dari dashboard.
10. Jangan mengekspos API key Groq atau Gemini pada frontend.
11. Gunakan bahasa Indonesia untuk isi dummy, dashboard, validasi, notifikasi, dan antarmuka awal.
12. Gunakan penamaan kode, class, function, database field, dan variable dalam bahasa Inggris.
13. Gunakan TypeScript secara konsisten dan hindari penggunaan `any` kecuali benar-benar diperlukan.
14. Gunakan komponen reusable dan hindari duplikasi kode.
15. Jangan membuat desain generik yang terlihat seperti template admin biasa. Website publik harus memiliki identitas visual modern, interaktif, dan profesional.
16. Dashboard admin harus lebih mengutamakan kemudahan penggunaan daripada animasi.
17. Semua perubahan penting harus tetap mengikuti dokumen ini sebagai sumber utama kebutuhan produk.

# 18. Chatbot

## 18.1 Tujuan

Chatbot membantu pengunjung mengenal:

* Profil Aditya
* Skill
* Pengalaman
* Pendidikan
* Proyek
* Sertifikasi
* Status ketersediaan
* CV
* Kontak

Chatbot tidak digunakan untuk menjawab topik umum di luar portofolio.

## 18.2 Provider

```text
Primary provider  : Groq
Fallback provider : Gemini
Final fallback    : FAQ dan navigasi statis
```

## 18.3 Alur Chatbot

```text
Pengguna mengirim pesan
→ Validasi input
→ Periksa intent sederhana
→ Cari FAQ
→ Cari data langsung dari database
→ Jika jawaban dapat dibuat tanpa AI, jawab langsung
→ Jika perlu analisis, susun context portofolio
→ Kirim ke Groq
→ Jika Groq gagal, kirim ke Gemini
→ Jika keduanya gagal, gunakan fallback statis
→ Simpan log teknis seperlunya
```

## 18.4 Pertanyaan yang Tidak Memerlukan AI

Contoh:

```text
Download CV
Tampilkan proyek Laravel
Apa email Aditya?
Apa skill frontend Aditya?
Tampilkan pengalaman terbaru
```

Pertanyaan tersebut dijawab langsung dari database.

## 18.5 Pertanyaan yang Menggunakan AI

Contoh:

```text
Apakah Aditya cocok untuk posisi junior full-stack developer?
Apa proyek Aditya yang paling relevan untuk posisi AI Engineer?
Rangkum pengalaman Aditya dalam tiga kalimat.
```

Laravel harus mengambil data relevan terlebih dahulu, kemudian mengirimkannya sebagai context kepada AI.

## 18.6 Aturan Jawaban

System prompt harus mengatur bahwa chatbot:

* Hanya menjawab berdasarkan data portofolio.
* Tidak mengarang proyek, skill, pengalaman, atau pencapaian.
* Tidak memberikan informasi sensitif.
* Menjawab singkat, ramah, dan mudah dipahami.
* Mengarahkan ke halaman proyek, CV, atau kontak jika relevan.
* Menolak pertanyaan di luar konteks dengan sopan.
* Tidak menyebut API key.
* Tidak perlu menyebut provider yang sedang digunakan kepada pengguna.

Contoh penolakan:

```text
Maaf, saya hanya dapat membantu menjawab pertanyaan mengenai
profil, kemampuan, pengalaman, dan proyek Aditya.
```

## 18.7 Fitur UI Chatbot

* Floating chat button
* Chat window
* Header
* Status
* Welcome message
* Suggested questions
* Message bubble
* Typing indicator
* Input
* Send button
* Error state
* Offline state
* Clear conversation
* Mobile fullscreen atau near-fullscreen
* Tombol menuju proyek
* Tombol download CV
* Tombol kontak

Gunakan Motion for React untuk animasi chatbot.

## 18.8 Rate Limit

Gunakan batas awal:

```text
5 sampai 10 pesan per menit per session atau pengunjung
Maksimal 500 karakter per pertanyaan
Maksimal jawaban sekitar 400 output token
Timeout AI sekitar 10 sampai 15 detik
Riwayat yang dikirim maksimal 6 sampai 10 pesan terakhir
```

## 18.9 Error yang Memicu Gemini

Fallback ke Gemini jika Groq mengalami:

* HTTP 429
* HTTP 500
* HTTP 502
* HTTP 503
* Timeout
* Respons kosong
* Respons tidak valid

Jangan menggunakan Gemini sebagai fallback untuk request invalid atau kesalahan validasi aplikasi.

## 18.10 Environment Variable

Gunakan konfigurasi seperti:

```env
AI_PRIMARY_PROVIDER=groq
AI_FALLBACK_PROVIDER=gemini

GROQ_API_KEY=
GROQ_MODEL=

GEMINI_API_KEY=
GEMINI_MODEL=

AI_TIMEOUT=15
AI_MAX_OUTPUT_TOKENS=400
```

Jangan mengirimkan API key ke frontend.

## 18.11 Service Structure

Gunakan struktur:

```text
app/
├── Contracts/
│   └── AiProviderInterface.php
├── Services/
│   └── AI/
│       ├── GroqProvider.php
│       ├── GeminiProvider.php
│       ├── AiRouterService.php
│       └── PortfolioContextService.php
└── Http/Controllers/
    └── ChatbotController.php
```

Setiap provider harus menghasilkan format respons internal yang seragam.

# 19. Animasi

## 19.1 GSAP

Gunakan GSAP untuk:

* Hero opening timeline
* Text reveal
* Parallax
* ScrollTrigger
* Experience timeline
* Project showcase
* Section pinning jika benar-benar diperlukan
* Horizontal motion secukupnya

Gunakan `@gsap/react` dan lakukan cleanup dengan benar.

## 19.2 Motion for React

Gunakan Motion untuk:

* Hover card
* Button interaction
* Mobile menu
* Modal
* Filter proyek
* Layout transition
* Toast
* Chatbot
* Loading
* Page transition
* Accordion

## 19.3 Aturan Animasi

* Jangan menggunakan GSAP dan Motion untuk properti yang sama pada elemen yang sama.
* Gunakan GSAP pada wrapper dan Motion pada child jika keduanya diperlukan.
* Jangan membuat semua elemen bergerak.
* Kurangi animasi pada mobile.
* Dukung `prefers-reduced-motion`.
* Animasi tidak boleh menghalangi interaksi.
* Hindari scroll hijacking yang mengganggu.
* Hindari intro panjang yang harus ditunggu pengguna.
* Jangan membuat animasi menyebabkan layout shift.

Durasi acuan:

```text
Hover kecil    : 150–250ms
Modal dan menu : 250–350ms
Section reveal : 500–700ms
Hero sequence  : 700–1200ms
```
